## Project

**Osbaldida 2026** — a mobile-first web app for a cycling-themed bachelor party event (5–7 June 2026, ~18 participants).

The app manages live scoring classifications, jersey competitions, event roadbook, rules, and a photo gallery.
It is used almost exclusively from mobile phones during the event.

## Commands

```bash
npm run dev                                                    # local dev server (Vite + Cloudflare bindings proxy)
npm run build                                                  # Astro build → dist/server/ + dist/client/
npm run preview                                                # preview production build locally (wrangler dev)
wrangler deploy --config dist/server/wrangler.json            # deploy to Cloudflare Workers
wrangler d1 migrations apply osbaldida-2026 --local           # apply DB migrations locally
wrangler d1 migrations apply osbaldida-2026 --remote          # apply DB migrations to production
```

Build output: `dist/server/` (Worker entry + wrangler.json) and `dist/client/` (static assets).
The `dist/server/wrangler.json` is generated by the adapter and merges `wrangler.toml` with adapter config.
Do not edit `dist/server/wrangler.json` directly — it is regenerated on every build.

## Core technology

- **Astro** with `@astrojs/cloudflare` adapter (SSR on Workers)
- **React islands** only where interactivity is needed (tab switchers, upload forms, scoring forms)
- **TypeScript** strict mode
- **Tailwind CSS** with custom cycling identity tokens
- **Cloudflare Workers** — runtime
- **Cloudflare D1** — all structured data (participants, scores, votes, photos metadata)
- **Cloudflare R2** — photo uploads
- **Zod** — all API input validation

## Architecture

```
src/
  pages/              # Astro pages (SSR) + api/ endpoints
  components/
    layout/           # Layout.astro, base shell
    navigation/       # BottomNav (mobile), TopNav (desktop)
    ui/               # Generic: buttons, cards, badges, chips
    jerseys/          # Jersey-specific display components
    teams/            # Team badge components
  features/
    classification/   # Ranking tables, tab switcher
    scoring/          # Admin score entry forms
    roadbook/         # Stage cards, timeline
    rules/            # Accordion rules sections
    gallery/          # Upload form, photo grid
    participants/     # Participant list, team grid
  server/
    db/
      client.ts       # D1 client wrapper
      migrations/     # SQL migration files
      seed.ts         # Seed data (teams, stages, climbs, participants)
    repositories/     # One file per entity — raw D1 queries only
    services/         # Business logic (classification ranking, scoring calc, auth)
    validation/       # Zod schemas for API inputs
    storage/          # R2 helpers for photo upload/read
  data/               # Static constants (event config, team colors, climb data)
```

Server data flow: `Astro page → service → repository → env.DB (D1)`. No raw SQL outside `repositories/`.

Full visual system: `docs/design-reference.md`.

## Auth

No full auth system. `/puntuaciones` uses a single shared code stored as `ADMIN_CODE` env var.
On entry, the server creates a session token (hash stored in D1 `admin_sessions` table) returned as an HTTP-only cookie.
Photo upload uses an optional `GALLERY_CODE`. Public pages have no auth.

## Key constraints

- All classification columns must fit within **360px** without horizontal scroll — no `overflow-x: auto` on tables.
- Minimum tap target: **44×44px**.
- Input `font-size` must be **≥ 16px** to prevent iOS auto-zoom.
- Bottom nav must include `padding-bottom: env(safe-area-inset-bottom)` for iPhone home bar.
- No external services or runtime dependencies beyond Cloudflare platform.

## Working style

1. Re-read the relevant `docs/` file.
2. Summarize the proposed change.
3. Ask for confirmation.

When uncertain about scoring rules, check `docs/scoring-system.md` before guessing.
When uncertain about logistics or roadbook content, check `reference/plan/planning.md` before guessing.

Saturday climb scoring (Stage 2) uses manual point entry — no fixed drink table rules. See `docs/scoring-system.md`.
Team-to-participant assignments are done via `/puntuaciones → Equipos — asignación` before the event.
